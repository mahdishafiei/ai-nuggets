# S2F-agent — Skill-Grounded Orchestration for the Sequence-to-Function Foundation Model Mess

Paper link: https://www.biorxiv.org/content/10.64898/2026.05.13.724757v1

## Script

Welcome back to Biomedical Agentic AI. Two episodes for you today — yesterday's bonus and now today's main item — because there's a second preprint on bioRxiv this week that's worth its own treatment. The earlier one, Talk2QSP from Sanofi, was about agents doing translation work in front of mechanistic pharmacology models. Today's paper is the same family of idea, but pointed at a different mess. It's called S2F-agent, posted to bioRxiv on the 15th out of a group at Zhejiang University, and the problem it solves is one that anyone who's actually tried to use the new wave of sequence-to-function foundation models in a real project will recognize immediately.

Let me set the stage. Sequence-to-function foundation models are the genomics-side answer to the protein structure prediction story. The idea is that you train a very large model on raw DNA, and it learns to predict the functional consequences of any base-pair change — chromatin accessibility here, RNA-seq read counts there, splicing, transcription factor binding, the works. AlphaGenome from DeepMind is one. Borzoi out of Calico is another. Evo 2 from the Arc Institute is a third. There are now more than a dozen of these models, each with its own pretrained checkpoint, its own input conventions, its own output heads, and its own runtime requirements. For a working computational biologist, that ecosystem is brilliant in theory and hostile in practice. To use AlphaGenome you tokenize one way. To use Borzoi you tokenize another way, on a different reference assembly, with a different strand convention. The outputs come back in incompatible coordinate systems. Some models run only on TPUs, some need specific GPU memory profiles, some have inference scripts that haven't been touched since the original release and rot on every CUDA upgrade. The fragmented ecosystem isn't a small annoyance — it's the dominant cost of using any of these models at all.

So the natural reaction in 2026 is: just throw a coding agent at it. Claude or GPT plus a Python environment plus the model repositories — let the agent figure it out. The S2F-agent authors are direct about why that fails. A general-purpose coding agent will happily wire up the wrong tokenization, point the model at the wrong reference assembly, mis-handle strand orientation, and produce a prediction that looks plausible and is silently meaningless. The constraint isn't reasoning capability; it's that the agent has no enforced model of what's biologically valid for each tool. So they build a system that has one.

The design has three pieces and the architecture itself is the part to study. First, canonical input keys. Every analysis the system can run is described in terms of a fixed vocabulary of biological inputs — a genomic region, a variant, a tissue, a cell type, an assay — with strict types and units. The user's free-text query gets translated into that canonical form once, up front, and everything downstream consumes the canonical form. Second, task-specific playbooks. For each kind of analysis — predict variant effect on chromatin accessibility, predict tissue-specific gene expression, score a regulatory element, and so on — there's an explicit playbook that says which model to call, with which input transformation, against which reference, returning which output normalization. The playbook is the domain expert's knowledge made executable. Third, normalized output contracts. Whatever model the playbook routes to, the returned object conforms to a single schema, so downstream comparison, aggregation, and visualization work uniformly across the eleven models the system currently integrates. AlphaGenome, Borzoi, Evo 2, and eight others — all behind a single interface.

The right way to think about this paper is that it's an existence proof for a pattern. Skill grounding is the term they use, and it's borrowed in spirit from the robotics agents literature, where you don't ask the manipulator to reason about hand geometry from scratch every action — you give it a library of grounded skills with preconditions and effects, and the planner composes them. S2F-agent is that, applied to a genomics tool ecosystem. The agent does the routing and the natural-language interpretation, but it never has to invent the right way to call a model — that's compiled into the playbook. When the playbook is missing or ambiguous, the system surfaces that, rather than confabulating.

Two things I want to draw out. The first is the routing-and-groundedness evaluation framing. The authors don't claim end-to-end accuracy on a benchmark of biological discoveries — that would be both impossible to define and easy to fake. They evaluate whether the system routes queries to the right model, whether the model is invoked with valid inputs, and whether the answer is grounded in what the model actually produced rather than what the language layer hallucinated. Those are the questions that matter for a tool layer. If you nail routing and groundedness, the underlying foundation models do the science. If you don't, no amount of polish at the language layer helps you.

The second point is the broader trajectory. There's a recurring pattern in agentic biomedical AI right now where the most useful agents are not the ones doing autonomous reasoning. They're the ones absorbing the fragmentation tax. Talk2QSP yesterday was about literature-to-simulation translation. S2F-agent today is about query-to-foundation-model orchestration. Both of these are duct tape work — but it's the duct tape that determines whether the underlying capabilities actually get used. If you're doing sequence-to-function analysis at any scale, this is the right shape of system to be building toward, and S2F-agent is one of the cleaner public templates for it. Worth reading the playbook section in particular.

That's the second episode for today. Back tomorrow with a fresh one.
