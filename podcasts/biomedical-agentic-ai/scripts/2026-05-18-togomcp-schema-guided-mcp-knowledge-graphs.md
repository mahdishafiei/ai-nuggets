# TogoMCP — Schema as Context Beats Orchestration as Logic for Life-Science Knowledge Graph Querying

Paper link: https://www.biorxiv.org/content/10.64898/2026.03.19.713030v2

## Script

Welcome back to Biomedical Agentic AI. Today's paper is one I have been waiting for, because it sits exactly on the intersection of three things this show cares about — knowledge graphs, the Model Context Protocol, and the question of how much engineering effort you actually need to wrap around a language model before it does something useful. It posted to bioRxiv yesterday as a version-two update from the Database Center for Life Science in Japan, and it is called TogoMCP.

The setup is familiar to anyone who has ever tried to write a SPARQL query against a serious biomedical knowledge graph. The center maintains an R-D-F portal that aggregates roughly sixty life-science databases under one endpoint — gene, protein, disease, drug, ortholog, variant, pathway, the usual cast. To get an answer out of it you need to know SPARQL and you need to know the schema of whichever database you are querying, because every database in the portal uses its own predicates, its own identifier conventions, its own naming for the same biological concept. That combination puts the resource out of reach of basically every working biologist who is not also a semantic-web specialist. So in principle you point a language model at the portal and ask it to do the translation for you. In practice, without schema-level context, the model invents predicates that do not exist, fails to map gene symbols to the right database-specific identifier, and you get a confidently wrong query that returns nothing.

TogoMCP attacks this in what I think is the right way. It recasts the language model as a protocol-driven inference engine that orchestrates a small set of specialized tools through the Model Context Protocol, and it does two specific things that turn out to matter a lot. The first is what they call an M-I-E file, short for metadata-interoperability-exchange. This is just a short YAML document, one per target database, that hands the model the structural and semantic context for that database at query time — the predicates that actually exist, the identifier shape, the join points. Not a full schema dump, not a hundred-page documentation paste, just enough context to make the next SPARQL generation grounded. The second is a two-stage workflow. Entity resolution happens first, as a separate step, through external rest endpoints that map a name like B-R-C-A-1 to the database-specific identifier the portal actually uses. Schema-guided SPARQL generation happens second, with the resolved identifiers already in hand. Those two stages do not get blended into one prompt.

The benchmark is fifty biologically grounded questions spanning five question types and twenty-three databases. Against an unaided baseline language model, TogoMCP shows a Cohen's d of one point eight two — a very large effect — with a Wilcoxon p-value well under a thousandth. Win rates exceed eighty percent on the question types that have precise, verifiable answers. So far this is the kind of result you would expect from a well-engineered retrieval-augmented system; the real interest is in the ablation.

The ablation pulls out the M-I-E file and the procedural protocol separately and asks which one is doing the work. The answer is that the schema files do most of the heavy lifting. Removing the M-I-E content costs about a half-point of mean per-question score, the bootstrap interval excludes zero, and a single one-line instruction telling the model to load the relevant schema file recovers almost the entire improvement that the full procedural protocol delivers. The protocol — the elaborate orchestration logic — does not move the mean. What it does is narrow the variance. Loss rate, the fraction of questions where the system actively underperformed the unaided baseline, drops from about five percent to under two percent when the procedural guidance is present.

That decomposition is the result I want listeners to walk away with, because it generalizes well past this specific paper. The conventional wisdom in agentic AI right now is that you need elaborate orchestration — careful tool routing, multi-step plans, supervisor loops — to get a language model to behave on a structured task. TogoMCP says, at least for knowledge-graph querying, that the orchestration is the safety net, not the engine. The engine is concise, dynamically-delivered schema context. If the model is told, at the right moment, what predicates exist and what the identifiers look like, it will write a correct SPARQL query without being walked through a flowchart. Procedural scaffolding still earns its keep by clipping the tail, but it is the wrong place to spend engineering effort if you are trying to lift the mean.

For the bigger picture — and this is the part I think Andrew will appreciate — the broader R-D-F open-data ecosystem in life sciences has spent a decade producing standards that nobody outside the semantic-web community could use directly. The bet behind TogoMCP is that you do not have to throw any of that work away to get to natural language as the user interface. You just need a thin per-database schema digest, delivered through the Model Context Protocol, and an agent that knows when to call entity resolution before query generation. That is a refreshingly small ask compared to "rebuild everything as a vector database."

Two caveats. The questions in the benchmark are biologically grounded but they are author-curated, and benchmarks of language-model agents on author-curated questions tend to look better than the same agents on user-generated questions in the wild. The other caveat is that this is still SPARQL on the back end, which means the ceiling on what TogoMCP can answer is set by what the federated graph actually knows. If a relationship is not in the R-D-F portal, no amount of schema context will conjure it.

Net of that, this is the cleanest argument I have seen this year for delivering structured-data context to agents through M-C-P rather than wrapping models in orchestration. If you are building anything that needs language models to talk to biomedical knowledge graphs, this is the architecture to copy. Back tomorrow with another one.
